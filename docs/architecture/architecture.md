# `src/` Architecture

This document is the source of truth for how `src/` is organized. Use it as a map when planning changes, writing new code, or reviewing PRs. If a change makes this stale, update this file in the same PR.

## Top-level layout

```
src/
  app/         Next.js App Router entrypoints (routes, layouts, error boundary, route handlers)
  inngest/     Background jobs: client, event registry/types, function definitions
  lib/         Framework-agnostic, reusable building blocks (see "src/lib" below)
  modules/     Feature modules — server (tRPC procedures) + ui (components, views) per domain
  trpc/        tRPC wiring: init, root router, server/client adapters, query client
  ui/          Cross-feature presentation layer: shadcn UI primitives, shared hooks, logo asset
  generated/   Prisma client output (do not edit; regenerated by `prisma generate`)
```

### Direction of dependencies

```
app  →  modules, trpc, inngest, ui, lib
modules → trpc, ui, lib            (server side may also use inngest client)
inngest → lib                       (NEVER imports from app/modules/ui)
trpc → modules, lib
ui   → lib                          (presentation only)
lib  → lib only                     (NEVER imports from app/modules/inngest/trpc/ui)
```

`src/lib/*` is the leaf layer. Reverse imports are forbidden — see "Import rules" below.

### Path alias

`tsconfig.json` maps `@/*` → `./src/*`. Always import via `@/...` rather than relative paths that climb out of a folder.

## `src/app/` — Next.js routes

App Router. Route groups live alongside the routes they namespace.

- `(home)/` — marketing/landing route group with its own layout.
- `projects/[projectId]/page.tsx` — authenticated project workspace.
- `api/inngest/route.ts` — Inngest webhook handler.
- `api/trpc/[trpc]/route.ts` — tRPC HTTP adapter.
- `layout.tsx`, `error.tsx`, `globals.css` — root chrome and global styles.

Pages should stay thin: import views from `src/modules/<domain>/ui/views` and procedures via tRPC. Keep data fetching in modules, not in route files.

## `src/modules/` — feature modules

One folder per product domain. Each module is internally split into `server/` and `ui/` so server-only code never leaks into client bundles.

```
modules/<domain>/
  server/
    procedures.ts         tRPC procedures for this domain
    <helper>.ts(.test.ts) Domain-specific server logic + colocated tests
  ui/
    components/           Domain components (presentational + container)
    views/                Page-level compositions (used by app/ routes)
```

Current modules:

- `home/` — UI only (project list, templates, form). No server procedures.
- `messages/` — server procedures + chat UI (mode selector, etc.).
- `projects/` — server procedures (incl. `naming.ts` for AI-generated project names) + workspace UI (assistant message, file explorer, fragment web view, header, thoughts modal, project view).
- `providers/` — server-only procedures exposing provider availability to the client. No UI.

Module → module imports should stay rare; if two modules need the same logic, lift it into `src/lib/`.

## `src/trpc/` — tRPC wiring

- `init.ts` — context + procedure builders.
- `routers/_app.ts` — root router that mounts module routers.
- `server.tsx` / `client.tsx` — server-component caller and client-side React hooks.
- `query-client.ts` — shared React Query client config.

When adding a new module router, register it in `routers/_app.ts`. Do not put business logic here.

## `src/inngest/` — background jobs

- `client.ts` — Inngest client.
- `events.ts` + `events.test.ts` — typed event registry. Add new event names/types here so producers and consumers share the same shape.
- `functions.ts` — Inngest function definitions (the agent run, etc.). These orchestrate `src/lib/agents`, `src/lib/sandbox`, and `src/lib/models` but contain no pure logic of their own — keep that in `src/lib/`.

## `src/ui/` — cross-feature presentation

Anything UI that is not tied to a single module lives here.

- `components/ui/` — shadcn primitives (button, dialog, sidebar, …). Generated by the shadcn CLI; the alias in `components.json` points here.
- `components/code-view/` — syntax-highlighted code viewer + theme CSS.
- `components/logo/logo.tsx` — logo component.
- `hooks/use-mobile.ts` — viewport hook.
- `logo/logo_1.svg` — logo asset.

Module-specific components do **not** belong here — put them under `src/modules/<domain>/ui/components/`.

## `src/lib/` — framework-agnostic building blocks

`src/lib` houses reusable logic with no awareness of Next, Inngest, HTTP, or React rendering. **Nothing in `src/lib` may import from `src/inngest`, `src/app`, `src/modules`, `src/trpc`, or `src/ui`.** Anything else in the repo can import from `src/lib`.

### Folder convention

Every concern under `src/lib/<concern>/` follows the same shape:

| File                | Purpose                                                                                 |
| ------------------- | --------------------------------------------------------------------------------------- |
| `constants.ts`      | Module-wide constants, default config, regex literals, lookup tables                    |
| `types.ts`          | Pure TypeScript types and interfaces (non-zod)                                          |
| `schemas.ts`        | Zod schemas + their inferred types (only when zod is used)                              |
| `<concern>.ts`      | One file per concern (verb- or noun-named: `state`, `factory`, `connect`, `preview`, …) |
| `<concern>.test.ts` | Vitest tests co-located with the concern they cover                                     |
| `index.ts`          | Barrel: `export * from "./<file>";` for each sibling. External imports use the barrel.  |

**Why this shape:**

- **Predictable navigation.** Every folder answers the same questions in the same place.
- **Stable import surface.** Consumers import from `@/lib/<concern>` (the barrel), not deep paths. Internal restructuring inside a folder does not break callers.
- **Cycle-resistant.** `constants` depends on `types` and nothing else; concern files depend on `constants`/`types`/`schemas`. No concern file imports a sibling concern unless one is a leaf utility (e.g. `tools.ts` uses `edits.ts`).

### Folders

#### `src/lib/agents/` — coding-agent runtime

Pure logic for the planner/executor agent loop. No awareness of inngest, HTTP, or where it runs.

- `constants.ts` — `AGENT_CONFIG` (token/step caps), regex for inferring verification kind, `DEFAULT_VERIFICATION_COMMAND`, `TASK_SUMMARY_RE`.
- `types.ts` — `RunState`, `EscalateDecision`, `Edit`, `EditResult`, `UsageTotals`, `PersistedTelemetry`, `TelemetryPayload`, `TelemetryStore`.
- `schemas.ts` — Zod schemas for planner and final-output IO (`PlanOutputSchema`, `FinalOutputSchema`, `VerificationRecordSchema`) + inferred types.
- `state.ts` — Run-state lifecycle: `createRunState`, `markVerification`, `hasSuccessfulVerification`, `inferVerificationKind`.
- `decisions.ts` — Pure decision helpers: `stepTextOf`, `extractTaskSummary`, `shouldEscalate`. No IO.
- `edits.ts` — Search/replace primitives used by tools: `applyEdit`, `EDIT_SCHEMA`, `truncateTo`, `exceedsLimit`.
- `telemetry.ts` — Telemetry assembly + persistence (`buildTelemetry`, `persistTelemetry`, `persistTelemetryWith`). Persistence accepts an injected `TelemetryStore` for testability.
- `tools.ts` — AI SDK `tool()` factories (`createTerminalTool`, `createReadFilesTool`, `createWriteFilesTool`, `createReplaceInFileTool`, `createApplyPatchTool`, `createRunBuild/Tests/LintTool`, `createFinalizeTool`). Each tool takes `{ getSandbox, runState }` deps so the caller controls where commands run.

#### `src/lib/models/` — model registry & resolution

- `constants.ts` — `MODEL_REGISTRY` (planner + executor specs from env) and `EXECUTOR_LADDER` (ordered fallback chain).
- `types.ts` — `ModelSpec`, `ResolvedModelConfig`, `KeyResolver`, `MessageRow`.
- `factory.ts` — `createModelProvider` (OpenRouter binding), `resolveSpec`/`resolveSpecWith`, `resolvePlannerModel`.
- `messages.ts` — Conversion + history retrieval for AI SDK message format (`toModelMessages`, `getPreviousMessages`).

#### `src/lib/errors/` — provider error classification

- `types.ts` — `ProviderErrorCategory`, `ClassifiedProviderError`, `ProviderErrorRule`.
- `constants.ts` — `PROVIDER_ERROR_RULES` (ordered match list; first match wins).
- `provider.ts` — `classifyProviderError(err)` for retryable/user-facing decisions on upstream provider failures.

#### `src/lib/sandbox/` — E2B sandbox lifecycle

- `constants.ts` — Timeouts, ports, preview server command, process probe command.
- `types.ts` — Structural interfaces (`SandboxConnection`, `PreviewSandboxConnection`, `SandboxClient`) so consumers can stub for tests without pulling in the E2B SDK.
- `connect.ts` — `connectSandbox`, `getSandbox`. Optional injected `SandboxClient` for tests.
- `preview.ts` — `getSandboxUrl`, `probePreviewOnce`, `isPreviewProcessRunning`, `ensurePreviewReady` (poll-or-start preview server).

#### `src/lib/providers/` — LLM provider catalog

- `types.ts` — `Provider` literal union.
- `config.ts` (`server-only`) — Reads API keys from env. `getProviderKey`, `isProviderAvailable`, `getProviderAvailabilityMap`.

#### `src/lib/rate-limit/` — TRPC rate limiting

- `types.ts` — `RateLimiter`, `RateLimitConfig` (the limiter is structural; tests pass a fake).
- `constants.ts` — Window/points defaults, `DEFAULT_RATE_LIMIT_CONFIG`, `GLOBAL_FALLBACK_KEY`.
- `factory.ts` — `createRateLimiter` (Prisma-backed).
- `consume.ts` — `hashKey`, `consume(limiter, rawKey)` (throws `TRPCError TOO_MANY_REQUESTS`), `consumeRateLimit(ip)` (no-op in dev). Holds the memoized default limiter as module-private state.

#### `src/lib/db/` — Prisma client singleton

- `index.ts` — Memoized `prisma` client. Dev mode caches it on `globalThis` to survive HMR; production constructs once. Replaces the legacy `src/db.ts`.

#### `src/lib/shared/` — cross-cutting helpers

- `utils/cn.ts` — `cn(...inputs)` Tailwind class merger (clsx + tailwind-merge). The `utils` alias in `components.json` points at `@/lib/shared/utils`, so shadcn-generated components resolve `cn` here.
- `utils/index.ts` — barrel.
- `test/stubs/server-only.ts` — Vitest stub for the `server-only` package so unit tests can import server modules without the bundler guard tripping.

#### Other folders following the same convention

- `src/lib/config/` — `env.ts` (typed env access, `isProduction`), `models.ts` (model id constants).
- `src/lib/log/` — Structured logger (`createLogger`, `timed`) plus `normalize.ts`/`schema.ts`.
- `src/lib/prompts/` — Prompt strings (`planner`, `executor`, `ask`, `naming`, `sections`, `assemble`) and provider cache options.
- `src/lib/schemas/` — Cross-cutting zod schemas (e.g. `thought`).

### Import rules

- **External callers always import from the folder barrel** (`@/lib/agents`, `@/lib/models`, etc.), not deep paths. Within a folder, sibling files import each other relatively (`./constants`, `./types`).
- **Direction of dependencies inside a concern**: `constants` and `types` are leaves. `schemas` is a leaf. Concern files depend on those, plus optionally on a sibling leaf utility (e.g. `tools.ts` → `edits.ts`).
- **`src/lib/*` MUST NOT import from `src/inngest`, `src/app`, `src/modules`, `src/trpc`, or `src/ui`.** Reverse direction is allowed.
- **`server-only` modules** (currently `providers/config.ts`) keep that pragma at the top — re-exporting them through a barrel is fine; the bundler enforces server boundaries.

## Where to put new code

| New thing                         | Folder                                                        |
| --------------------------------- | ------------------------------------------------------------- |
| New page / route                  | `src/app/<segment>/page.tsx` (+ layout if it has chrome)      |
| New tRPC procedure for a domain   | `src/modules/<domain>/server/procedures.ts`                   |
| New domain UI component           | `src/modules/<domain>/ui/components/`                         |
| New page-level composition        | `src/modules/<domain>/ui/views/`                              |
| New shadcn primitive              | `src/ui/components/ui/` (run shadcn CLI; alias is configured) |
| New cross-feature hook            | `src/ui/hooks/`                                               |
| New Inngest event                 | `src/inngest/events.ts` (add type + register)                 |
| New Inngest function              | `src/inngest/functions.ts`                                    |
| New tool the agent can call       | `src/lib/agents/tools.ts`                                     |
| New escalation heuristic          | `src/lib/agents/decisions.ts`                                 |
| New telemetry field               | `src/lib/agents/{types,telemetry}.ts`                         |
| Add a model fallback rung         | `src/lib/models/constants.ts` (`EXECUTOR_LADDER`)             |
| New provider error category       | `src/lib/errors/{types,constants}.ts`                         |
| New sandbox capability            | `src/lib/sandbox/<concern>.ts`                                |
| New shared zod schema             | `src/lib/schemas/<name>.ts`                                   |
| New rate-limit policy             | `src/lib/rate-limit/{constants,factory}.ts`                   |
| New shared util (e.g. formatters) | `src/lib/shared/utils/<name>.ts` (export from `index.ts`)     |
| New test stub                     | `src/lib/shared/test/stubs/<name>.ts`                         |

If a new lib concern doesn't fit any existing folder, create `src/lib/<concern>/` with the same shape (`constants.ts`, `types.ts`, concern files, `index.ts`). Do not add files directly under `src/lib/`.

## Important root-level pointers

These live outside `src/` but materially affect how code in `src/` is structured. Update this section if any of them move.

- **`tsconfig.json`** — `@/*` path alias maps to `src/*`. Use it everywhere.
- **`components.json`** — shadcn config. Aliases: `components` → `@/ui/components`, `ui` → `@/ui/components/ui`, `hooks` → `@/ui/hooks`, `utils` → `@/lib/shared/utils`, `lib` → `@/lib`. Keep these in sync if folders move.
- **`vitest.config.ts`** — test runner config; resolves `@/*` and registers the `server-only` stub from `src/lib/shared/test/stubs`.
- **`next.config.ts`** — Next.js config.
- **`prisma/`** — Prisma schema and migrations. Generated client lands in `src/generated/prisma` (imported by `src/lib/db`).
- **`sandbox-templates/`** — E2B sandbox templates used by `src/lib/sandbox`.
- **`docs/`** — `architecture/` (this file), `documentation/` (long-form references), `plans/{open,completed}` (planning docs; new plans go in `open/`, finished ones move to `completed/`).
- **`Makefile`, `mprocs.yaml`, `.husky/`** — dev workflow (multi-process dev, git hooks). Touch only when changing local dev ergonomics.
